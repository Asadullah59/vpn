Features

  • Simple web interface

    All user and server configuration is done from a simple web interface that is protected by a username and password login.

  • User management

    Manage multiple organizations, users and vpn servers. Users certificates are automatically disabled when the user is deleted.

  • Anonymity

    Improve your online anonymity, all your internet traffic will appear to be coming from your vpn server this hides your public IP and the geolocation of your IP.

  • Security

    All traffic between your computer and the vpn server is encrypted. The traffic can't be monitored or exploited by your ISP/college or while on public wifi.

  • Two-step authentication

    Optional two-step authentication is available using Google Authenticator for improved security. Users will be required to enter a TOTP code when connecting.

  • Bypass Censors

    Bypass censored websites and blocked ports. Websites that block access to countries outside the US can also be accessed if your VPN server is located in the US.

Frequently Asked Questions

Q
Is the demo a real installation?
A
The demo uses javascript to emulate the functionality of the python server available in a real installation. All of the data is stored locally and is reset when the page is reloaded. Some functionality is missing from the demo
Q
Can the security of the vpn servers be improved?
A
By default the vpn servers will only generate 1536bit DH parameters, higher values can take a very long time depending on the availability of random data on the server. This can be increased by adding 
dh_param_bits=2048
or 
dh_param_bits=4096
to the conf file 
/etc/pritunl.conf
then restarting the pritunl service. This will only apply to servers created after restarting the service. Creating a new server can take several hours with higher values if there is limited random data available
Q
Can only lan traffic be routed trough the vpn server?
A
By default the servers will route all traffic trough the vpn tunnel. This can be changed in the server settings by selecting "Local Network Only" and entering the network address and subnet mask of the local network such as "192.168.0.0/24" or "10.0.0.0/8". This will route only traffic destined to the local network.
Q
What is the default login and password?
A
The default login and password is "admin"
Q
How can the login password be reset?
A
Edit 
/etc/pritunl.conf
and delete the line containing 
password=
then restart the pritunl service
Q
How can the users and servers be backed up?
A
All data is stored in 
/var/lib/pritunl
. A backup can be downloaded by logging into the web interface and opening 
https://<SERVER_ADDRESS>:9700/export
. To restore stop the pritunl service and remove the old files in 
/var/lib/pritunl
then extract the backup archive to the directory.
Q
What default ssl options are used?
A
The client and server keys are 4096bit RSA, SHA1 MD and 1536bit DH parameters. This can be changed by adding 
key_bits=4096
and 
dh_param_bits=1536
to the conf file 
/etc/pritunl.conf
Q
Where is the pritunl log file located?
A
The log file is located at 
/var/log/pritunl.log
Q
Where are the ssl certificates located?
A
All the certificates and conf files are stored in 
/var/lib/pritunl
Q
How can bugs be reported?
A
Bugs can be reported on the GitHub Issues page
Q
Can the user keys be downloaded with wget?
A
A button next to the users status can be used to generate a temporary url for the user key that can be downloaded without authenticating using an application like wget
Q
Do the openvpn servers autostart on reboot?
A
Yes, this can be disabled by adding 
auto_start_servers=false
to 
/etc/pritunl.conf
Q
How can a user be removed from the server?
A
Deleting a user will automatically restart any server's available to the user disconnecting the user and then disable the user's certificate
Q
When will user keys need to be redownloaded?
A
User keys will need to be updated after attaching the users organization to a server or when a servers public ip or port is changed
Q
How is the two-step authentication code validated?
A
Each time a user attempts to connect OpenVPN will pass the user and two-step authentication code to a python script. The script will validate the code and respond with an exit code indicating if the code was valid
Q
Can a two-step authentication code be used twice?
A
After a two-step authentication code has been validated it will be stored in a file and any further attempts to use the same code will be invalid
Q
Is the username needed with two-step authentication?
A
The username is not checked or used by the server and can be left blank, only the password is used with two-step authentication
Q
Are failed vpn authentication attempts logged?
A
All failed vpn authentication attempts of revoked user certs and invalid two-step authentication codes are logged to 
/var/lib/pritunl/auth.log
Q
Why does the server connect to ip.pritunl.com?
A
When the service starts an HTTP requests is sent to ip.pritunl.com to get the public IP address of the server. This is used to automatically fill in the public IP setting when creating a new server. The source code for the ip.pritunl.com server is available at github.com/zachhuff386/pritunl-ip. This can be disabled by adding 
get_public_ip=false
to 
/etc/pritunl.conf
Q
How can inline certificates be disabled?
A
By default the certificates and key for openvpn will be included in the user configuration file instead of having seperate files for the certificates and keys. This can be disabled by adding 
inline_certs=false
to 
/etc/pritunl.conf

REST API

GET /auth
Returns authentication status.
Response
authenticated True if authenticated otherwise false
Python Example
response = requests.get( 'https://localhost:9700/auth', ) assert(response.status_code == 200) print response.json() { 'authenticated': False, }
POST /auth/token
Returns authentication token required for making protected API calls. API token stays valid until server is restarted.
Parameters
username String username default is "admin"
password String password
Response
auth_token Authentication token
Python Example
response = requests.post( 'https://localhost:9700/auth/token', headers={'Content-Type': 'application/json'}, data=json.dumps({ 'username': 'admin', 'password': 'admin', }), ) assert(response.status_code == 200) print response.json() { 'auth_token': '264e30e7032b467bb0174b22bdb8ce13', }

GET /event/:last_event
Long polling request that returns a list of events that have occurred. Request will poll for up to 30 seconds. The current time should always be retrieved from a time event to avoid problems with out of sync clocks. A list of events can be found in 
constants.py
. New event IDs should be check against a list of events already recieved to avoid processing an event more then once.
URL Parameters
last_event Optional epoch time in milliseconds of last event. If left out a time event will be returned giving the current epoch time in milliseconds on the server
Parameters
auth_token Authentication token
Response
id Event ID
type Event type
resource_id Event resource ID
time Event epoch time in milliseconds
